## 如何用最小的代价做产品：MVP
[[#精益创业]]
**最小**：只需要用一种方法验证想法的可行性，软件开发只是验证想法的手段。可以用原型图、文档等方法去表示应用交互流，也可以去进行推演。

如何衡量产品设计的好不好？确保PM想明白了
- 如果不做，用户会怎样
- 有无可替代方案

**可行**：不是一个模块做得有多完整，而一条用户路径是否通畅。通过分解保证核心链路的体验。
比如做一个贷款软件，就先实现最重要的贷款 & 还款能力。因为借款最早一个月才会还，所以最早可以先开发贷款，后面再开发还款，也可以保证交付。

## 答疑
**面对不了解的技术如何分解项目**
做技术spike，关注这个项目在系统中的应用场景 + 关注点，是性能还是吞吐，也需要做一些实验验证。完事儿之后要把学习时候搞得原型代码丢掉，重新写一份。

不要混淆现状和目标：目标应该怎么做，现状是正在怎么做。正因为对现状不满意，所以才会探索改进方法。
假设现在不忙了，你打算怎么办？忙只是借口，不思考的借口。

## 谨慎地将工作自动化
**不要自动化**：做自动化之前，要先明确做的价值，如果不知道为什么做，做了有什么用，那不如不做，节省下精力本身也是价值。

**要自动化**：要将自己的工作尽可能自动化，比如说代码部署。不过现在已经挺智能了，不太需要我干啥了。工作中补充日志，细化异常打印也是自动化，降低了我们人工介入的成本。

**要懂软件设计**：软件设计是高层抽象，可变性较弱，实现的可变性就很强，先学会了软件设计，再学习框架和工具，就更容易。这本身也是懒惰的表现。

## 持续交付：将部署也纳入开发的考量

**持续集成**：仅考虑到开发环节，主要包含基本的检查，可以打出一个可发布的包
**持续交付**：持续集成 + 测试环境（自动） + 预生产环境（自动） + 生产环境（手动），让项目时刻处于可发布的状态。
有点像我们每次push都会自动化集成到staff环境，然后发布上去。

**DevOps**：自动化部署能力。

**扩展**：持续就需要自动化 + 验证 + 反馈，需要标准化的建立。而软件流程除了开发测试上线外，也包含了根据业务场景和用户行为选择更合适的业务策略等等，这些东西都没有标准化，所以我们的系统迭代本身也是手动验证，无法做到持续验证～

## 如何正确分层：构建好自己的领域模型
分层的本质是抽象。
以Java为例，常见的MVC三层架构，包含了Controller + Service + Dao层。
三层架构可以让我们屏蔽细节，专注于抽象的逻辑。

例如Controller负责了访问方式，比如RPC的接口层，http接口层都是外部服务访问内部逻辑的方式，从CS -> BS，从浏览器到移动端，都是接入方式的变化。

例如Dao层控制的数据存储，数据存储可以是MySQL、Oracle、文本文件、Redis等，本身也是可变化的。

Service层本身应该是构建好抽象供上下两层使用的，无论Controller和Dao如何变化，只需要适配一下就可以接入。
这种Service更应该叫做领域对象，具体这俩啥区别我也还不太清楚。细节可以学习下领域驱动设计。
如果定义好了抽象，那么系统就像一个六边形架构，只需要修改适配就可以用到其他地方。
![[Pasted image 20260201210344.png]]
领域模型，也可以抽象复用到其他领域，比如可以使用DSL语言定义～，这里没细说，但看起来是DDD那一块儿的东西。


## 用简单的技术解决问题，直到问题变得更复杂
淘宝本身也是一个很简单的系统，只要可以上线商品，用户能看到、能购买即可。
但是随着业务体量的扩大，现有的技术架构就会面临很多挑战。这是系统架构演进的根本动力。
所以我们要针对系统的量级特点去选择合适的技术，不要为了技术用技术。





## DDD是道，微服务是术
[[领域驱动设计]] 以后学学DDD啊Bro
DDD的本质是把技术从技术视角抽象为业务视角，进而适配变化的业务场景。
微服务本身是对划分后的系统进行单独的管理和部署迭代。边界划分才是最核心的东西。

如果想搞微服务，不如现在一个系统里边划分模块，单独迭代，等到演进的差不多了，再单独抽成微服务～
