# 基本原理
## Java如何在虚拟机运行

**JRE和JDK**

|     | 描述                                   |
| --- | ------------------------------------ |
| JRE | 仅包含运行Java程序的必需组件，包括Java虚拟机以及Java核心类库 |
| JDK | JRE + 一系列诊断工具                        |

**为什么要在虚拟机中运行**
1. 屏蔽底层系统差异，一次编写到处运行
2. 托管环境：代替我们处理一些代码中冗长而且容易出错的部分，比如垃圾回收。还有数组越界、动态类型、安全权限等等的动态检测

**虚拟机如何运行Java字节码**
- 虚拟机角度：虚拟机会先把编译的class文件存入虚拟机，具体会存在方法区里。在方法执行的时候会压入方法栈。空间可以不连续。
- 硬件角度：需要把字节码解释成机器码执行。具体有两种执行方法：解释执行（逐条将字节码翻译成机器码并执行）、JIT即时编译（一个方法中包含的所有字节码编译成机器码后再执行）。
	- 前者无需编译，后者效率更快。
	- 混合模式：把不常用的逐行执行，常用的就编译机器码，效率更高。

**JVM虚拟机效率**
混合模式中的JIT就可以很好的优化执行性能。
此外，JIT的机器码有更多的运行时信息，可以优化的比C++效率还高。比如面向对象的**虚方法**，会关联很多目标方法，但实际上只会用一个，就可以优化下。
具体咋优化的不清楚。


**HotSpot还有分层即时编译**

| 即时编译器 | 描述                                                                      |
| ----- | ----------------------------------------------------------------------- |
| C1    | C1又叫做Client编译器，面向的是对启动性能有要求的客户端GUI程序，采用的优化手段相对简单，因此编译时间较短               |
| C2    | C2又叫做Server编译器，面向的是对峰值性能有要求的服务器端程序，采用的优化手段相对复杂，因此编译时间较长，但同时生成代码的执行效率较高。 |
| Graal | Java10引入的，后面再说，TODO 补充一下                                                |
热点方法会先用C1编译，热点中的热点会用C2编译。
HotSpot的即时编译是在额外的线程搞的，不影响主流程。根据CPU核心数分配编译线程，C1和C2比例大部分在1:2。这样机器解释和编译可以一起搞，第二次执行的时候就可以用编译的东西了。

## Java的基本类型
**JVM中的boolean类型**
在Java中我们写吃过饭没 = true or false，在Java编译后就会映射为0 ，1。这是规范
但是编译好了之后，JVM只看0或者1了，如果有人给改成2他也看不出来。
```java
public class Foo {
  public static void main(String[] args) {
    boolean 吃过饭没 = 2; // 直接编译的话javac会报错，可以用编译工具偷偷改
    if (吃过饭没) System.out.println("吃了");
    if (true == 吃过饭没) System.out.println("真吃了");
  }
}
```
上面这个代码就会只打印第一个，不会打印第二个

**基本类型**
1. 基本类型：boolean、byte、short、char、int、long、float、double
2. 浮点和无穷：
    1. 浮点类型采用IEEE754，有+0.0F即0x0000000和-0.0F即0x8000000，尽管内存数值不同，但+0.0F == -0.0F是true
    2. 任意正浮点（不包括+0.0F）除以+0.0F得到正无穷即0x7F800000，任意正浮点（不包括+0.0F）除以-0.0F得到负无穷即0xFF800000
3. NaN：
    1. [0x7F800001, 0x7FFFFFFF]和[0xFF800001, 0xFFFFFFFF]对应的都是NaN
    2. 一般计算得到的NaN是0x7FC00000，称为标准NaN，其他NaN称为不标准NaN
    3. 对于NaN，除了“!=”始终返回true之外，所有其他比较结果都会返回false 

基本类型的大小
1. 解释器使用的解释栈帧有两个主要部分：局部变量区、字节码的操作数栈
    1. 局部变量区存放：普通意义的局部变量、实例方法的this指针、实例方法的实参
    2. 局部变量区等价于数组，32位jvm的单元长度为4byte，64位jvm的单元长度为8byte，long和double占用两个单元，其他基本类型和引用类型都占一个单元
    3. 以上的小整型占用一个字长的情况仅适用于局部变量，而在堆中的byte/char/short都是1/2/2个byte，即与它们的值域吻合
2. Boolean字段占用1byte，而boolean数组直接用byte数组实现。为保证堆中boolean合法，jvm存储时显式mask掉高位，只取最后一位
3. Jvm中的算数运算几乎都依赖操作数栈，需将堆中的boolean/byte/short/char加载到操作数栈，再在栈上当作int运算
    1. 对于boolean和char，加载时进行零扩展，高位填0
    2. 对于byte和short，加载时进行符号扩展，非负数高位填0，负数高位填1

## JVM如何加载Java类
| 类型   | 具体类  | 加载  | 链接  | 初始化 |
| ---- | ---- | --- | --- | --- |
| 基本类型 | -    | ❌   | ✅   | ✅   |
| 引用类型 | 类    | ✅   | ✅   | ✅   |
|      | 接口   | ✅   | ✅   | ✅   |
|      | 数组类  | ❌   | ✅   | ✅   |
|      | 泛型参数 | ❌   | ❌   | ❌   |
JVM加载类分成了加载、链接、初始化三步，其中并不是所有的类型都会走完全部的三步。
比如引用类型中的数组类，他是JVM直接生成的，无需加载。而类和接口都是用字节流加载到虚拟机里的，所以这种就需要加载。
基本类型也是无需加载的，JVM内部指令集已支持。
泛型参数后面会被擦除，所以实际上走的应该还是类或者接口的步骤。

综上，所有类都需要执行链接 + 初始化，是否需要加载取决于变量类型。

**加载**
启动类加载器（bootstrap class loader）：C++实现的，没有Java对象，只能用null代指。
- 加载最为基础、最为重要的类，比如存放在JRE的lib目录下jar包中的类（以及由虚拟机参数-Xbootclasspath指定的类。

扩展类加载器（extension class loader）：上面的子类，
- 加载相对次要、但又通用的类。如：存放在JRE的lib/ext目录下jar包中的类、由系统变量java.ext.dirs指定的类

应用类加载器（application class loader）：
- 负责加载应用程序路径下的类。如：虚拟机参数-cp/-classpath、系统变量java.class.path、环境变量CLASSPATH所指定的路径。）
- 应用程序中包含的类默认由应用类加载器加载

平台类加载器（platform class loader）：在Java9之后，扩展类加载器改成了这个。除了少数几个关键模块（如java.base）由启动类加载器加载，其他都是平台类加载器加载的。


PS：除了启动类加载器，其余类加载器都是java.lang.ClassLoader的子类，有对应的Java对象，需要加载到虚拟机里后，才能进行类加载。

类加载器的命名空间作用：在Java虚拟机中，类的唯一性是由类加载器实例以及类的全名一同确定的。即便是同一串字节流，经由不同的类加载器加载，也会得到两个不同的类。在大型应用中，我们往往借助这一特性，来运行同一个类的不同版本。

**链接**
- **验证**：确保被加载类满足Java虚拟机的约束条件
	- 通常，Java编译器生成的类文件都会满足Java虚拟机的约束条件，例外情况见TODO（字节码注入）
- **准备**：
	- 被加载类的静态字段分配内存，静态字段的初始化在初始化阶段进行
	- 构建和其他类层级相关的数据结构，比如虚方法表。
- **解析**：将符号引用解析为实际引用
	- 如果符号引用指向一个未被加载的类，或者未被加载类的字段或方法，那么解析将触发这个类的加载（但未必触发这个类的链接以及初始化）
	- **符号引用**：对于一个方法调用，编译器会生成一个包含目标方法所在类的名字、目标方法的名字、接收参数类型以及返回值类型的符号引用，来指代所要调用的方法。

Java虚拟机规范没要求在链接过程中完成解析。它仅规定了：如果某些字节码使用了符号引用，那么在执行这些字节码之前，需要完成对这些符号引用的解析。

所以我理解就是没用的时候可以先保留符号引用，真的要用的时候才会解析成实际引用

**初始化**
- 常量值（ConstantValue）赋值：被final修饰的基本类型或字符串，直接由虚拟机完成
- < clinit >方法：直接赋值操作，以及所有静态代码块中的代码，都会被放到这个方法里

初始化阶段会为标记为常量值的字段赋值 + 执行< clinit >方法。

Java虚拟机会通过加锁来确保类的< clinit >方法仅被执行一次，所以类初始化是线程安全的。
这个特点常用与实现单例模式，程序可以确保多线程环境下有且仅有一个Singleton实例。

触发类初始化的时机：
1. 当虚拟机启动时，初始化用户指定的主类；
2. 当遇到用以新建目标类实例的new指令时，初始化new指令的目标类；
3. 当遇到调用静态方法的指令时，初始化该静态方法所在的类；
4. 当遇到访问静态字段的指令时，初始化该静态字段所在的类；
5. 子类的初始化会触发父类的初始化；
6. 如果一个接口定义了default方法，那么直接实现或者间接实现该接口的类的初始化，会触发该接口的初始化；
7. 使用反射API对某个类进行反射调用时，初始化这个类；
8. 当初次调用MethodHandle实例时，初始化该MethodHandle指向的方法所在的类。

问题1：新建数组（第11行）会导致LazyHolder的加载吗？会导致它的初始化吗？
```bash
$ echo '
public class Singleton {
  private Singleton() {}
  private static class LazyHolder {
    static final Singleton INSTANCE = new Singleton();
    static {
      System.out.println("LazyHolder.");
    }
  }
  public static Object getInstance(boolean flag) {
    if (flag) return new LazyHolder[2];
    return LazyHolder.INSTANCE;
  }
  public static void main(String[] args) {
    getInstance(true);
    System.out.println("----");
    getInstance(false);
  }
}' > Singleton.java
$ javac Singleton.java
$ java -verbose:class Singleton
```
答：新建数组只会导致LazyHolder的**类加载**，但不会触发其**类初始化**。只有首次主动使用类（如访问静态字段/方法）时才会初始化。# 高效实现

问题2：新建数组会导致LazyHolder的链接吗？
```bash
$ java -cp /path/to/asmtools.jar org.openjdk.asmtools.jdis.Main Singleton$LazyHolder.class > Singleton$LazyHolder.jasm.1
$ awk 'NR==1,/stack 1/{sub(/stack 1/, "stack 0")} 1' Singleton$LazyHolder.jasm.1 > Singleton$LazyHolder.jasm
$ java -cp /path/to/asmtools.jar org.openjdk.asmtools.jasm.Main Singleton$LazyHolder.jasm
$ java -verbose:class Singleton
```
答：新建数组会触发`LazyHolder`的**加载**，但不会触发其**链接**或初始化。链接和初始化会延迟到首次主动使用类时（如访问静态字段/方法）。

### 重载与重写
**同一个类中的方法如何进行重载**
首先会按照**参数声明类型**选择重载方法，然后再选择**最贴切**的进行执行

如何根据传入**参数的声明类型**（注意与实际类型区分）来选取重载方法：
1. 在不考虑对基本类型自动装拆箱（auto-boxing，auto-unboxing），以及可变长参数的情况下选取重载方法；
2. 如果在第1个阶段中没有找到适配的方法，那么在允许自动装拆箱，但不允许可变长参数的情况下选取重载方法；
3. 如果在第2个阶段中没有找到适配的方法，那么在允许自动装拆箱以及可变长参数的情况下选取重载方法。

**判断贴切的依据**：形式参数类型的继承关系，我理解会更优先选择子类
开头的case中，第一个和第二个方法都很贴切，但是第二个方法的第一个参数是String，是Object的子类，所以他的贴切程度更高

**不同类的方法如何进行重载**
不同类一般都是继承关系，在父子类中有相似的方法
- 参数类型不同 + 同名：子类构成重载
- 参数类型相同 + 同名 + 静态方法：子类方法隐藏了父类方法
- 参数类型相同 + 同名 + 非静态方法 + 非私有：子类构成了重写


### JVM的静态绑定与动态绑定（没完成）
**Java虚拟机如何识别方法**
Java识别方法依赖类名、方法名以及方法描述符（method descriptor），JVM和Java的方法描述符有差异。
- Java的方法描述符：入参
- JVM的方法描述符：入参 + 出参

所以父子类中有**方法名 + 出入参完全一样**的**非私有、非静态**的方法，JVM才会把这个方法当作重写。

Java字节码中与调用相关的指令共有五种

| 指令名称            | 描述                                                   |
| --------------- | ---------------------------------------------------- |
| invokestatic    | 用于调用静态方法                                             |
| invokespecial   | 用于调用私有实例方法、构造器，以及使用super关键字调用父类的实例方法或构造器，和所实现接口的默认方法 |

下面这个case展示了前四种指令的调用情况
``` java
interface 客户 {
  boolean isVIP();
}

class 商户 {
  public double 折后价格(double 原价, 客户 某客户) {
    return 原价 * 0.8d;
  }
}

class 奸商 extends 商户 {
  @Override
  public double 折后价格(double 原价, 客户 某客户) {
    if (某客户.isVIP()) {                         // invokeinterface      
      return 原价 * 价格歧视();                    // invokestatic
    } else {
      return super.折后价格(原价, 某客户);          // invokespecial
    }
  }
  public static double 价格歧视() {
    // 咱们的杀熟算法太粗暴了，应该将客户城市作为随机数生成器的种子。
    return new Random()                          // invokespecial
           .nextDouble()                         // invokevirtual
           + 0.8d;
  }
}
```

Java虚拟机中的静态绑定指的是在解析时便能够直接识别目标方法的情况，而动态绑定则指的是需要在运行过程中根据调用者的动态类型来识别目标方法的情况。

### 基础知识
**GC Root**：初始的存活对象合集（live set），可以理解为由堆外指向堆内的引用。
包含如下四种
1. Java方法栈桢中的局部变量；
2. 已加载类的静态变量；
3. [[JNI Handles]]：强引用弱引用句柄啥的，不太懂。后面补充一波
4. 已启动且未停止的Java线程。

**引用计数法**：判断对象是否存活的方法。每个对象添加一个引用计数器，用来统计指向该对象的引用个数。一旦某个对象的引用计数器为0，则说明该对象已经死亡，便可以被回收了。
需要截获所有引用变更情况，用来更新引用计数器

**但是无法处理循环引用的对象，会导致对象无法回收造成内存泄漏**

**可达性分析**：判断对象存活的方法。可以从GCROOT开始探测所有对象，探测到了就会mark，最后没有被mark的就会被回收。
在多线程场景下可能会出现漏标或者误标的情况：
- 漏标：本来该标记存活的的没标记，导致被错误回收了，可能会导致系统异常。
- 误标：本来不该标记存活的标记了，导致没有回收掉，问题不大，可以下次再回收。


**Stop-the-world**：Stop-the-world指的是暂停其他线程，进行GC。这个机制是依赖**安全点**进行的。

**安全点**：指的是内存堆栈不会发生变化的时间段，而不是暂停其他线程的运行。这样系统就可以安全的进行可达性分析（应该也伴随着真实回收）


### 垃圾回收的三种方式
- 直接清除：直接清空内存，后面有需要再分配
	- 好处：实现简单
	- 坏处：内存碎片、分配效率低，需要遍历空闲空间列表
- 清理后压缩：把内存空间转移成连续空间
	- 好处：解决了碎片问题，分配效率也比较高
	- 坏处：压缩算法的性能损耗较大
- copy清理：堆空间一分为二，每次清理的时候都copy到另一个地方，copy过去就是连续的。
	- 好处：解决了内存碎片，分配效率高
	- 坏处：堆空间使用效率低，只能用一半


### 虚拟机堆划分
内存被分为新生代和老年代。
其中新生代分成了Eden区，以及两个大小相同的Survivor区，默认情况下采用动态分配策略，控制新生代内部的比例，也可以单独设置写死。

**TLAB**：为了避免多个线程申请同一个内存空间，可以预先为每个线程分配一波内存，并且通过并发控制避免重复分配

**晋升老年代**：每次GC的时候，存活的对象都会在两个Survivor之间来回copy，copy到一定次数就会把对象放到老年代，或者Survivor占用到一定比例的话，把复制次数较多的放到老年代。

**新生代内存回收（Young GC、Minor GC）**：使用标记 - 复制算法，因为大部分对象预期都要被删除 + 无需进行全堆扫描，所以效率比较高。
但是如果老年代中存在对新生代的引用，那必须进行全堆才能知道到底有没有用，如何解决？**卡表**
