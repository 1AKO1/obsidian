# 基本原理
## Java如何在虚拟机运行

**JRE和JDK**

|     | 描述                                   |
| --- | ------------------------------------ |
| JRE | 仅包含运行Java程序的必需组件，包括Java虚拟机以及Java核心类库 |
| JDK | JRE + 一系列诊断工具                        |

**为什么要在虚拟机中运行**
1. 屏蔽底层系统差异，一次编写到处运行
2. 托管环境：代替我们处理一些代码中冗长而且容易出错的部分，比如垃圾回收。还有数组越界、动态类型、安全权限等等的动态检测

**虚拟机如何运行Java字节码**
- 虚拟机角度：虚拟机会先把编译的class文件存入虚拟机，具体会存在方法区里。在方法执行的时候会压入方法栈。空间可以不连续。
- 硬件角度：需要把字节码解释成机器码执行。具体有两种执行方法：解释执行（逐条将字节码翻译成机器码并执行）、JIT即时编译（一个方法中包含的所有字节码编译成机器码后再执行）。
	- 前者无需编译，后者效率更快。
	- 混合模式：把不常用的逐行执行，常用的就编译机器码，效率更高。

**JVM虚拟机效率**
混合模式中的JIT就可以很好的优化执行性能。
此外，JIT的机器码有更多的运行时信息，可以优化的比C++效率还高。比如面向对象的**虚方法**，会关联很多目标方法，但实际上只会用一个，就可以优化下。
具体咋优化的不清楚。


**HotSpot还有分层即时编译**

| 即时编译器 | 描述                                                                      |
| ----- | ----------------------------------------------------------------------- |
| C1    | C1又叫做Client编译器，面向的是对启动性能有要求的客户端GUI程序，采用的优化手段相对简单，因此编译时间较短               |
| C2    | C2又叫做Server编译器，面向的是对峰值性能有要求的服务器端程序，采用的优化手段相对复杂，因此编译时间较长，但同时生成代码的执行效率较高。 |
| Graal | Java10引入的，后面再说，TODO 补充一下                                                |
热点方法会先用C1编译，热点中的热点会用C2编译。
HotSpot的即时编译是在额外的线程搞的，不影响主流程。根据CPU核心数分配编译线程，C1和C2比例大部分在1:2。这样机器解释和编译可以一起搞，第二次执行的时候就可以用编译的东西了。

## Java的基本类型
**JVM中的boolean类型**
在Java中我们写吃过饭没 = true or false，在Java编译后就会映射为0 ，1。这是规范
但是编译好了之后，JVM只看0或者1了，如果有人给改成2他也看不出来。
```java
public class Foo {
  public static void main(String[] args) {
    boolean 吃过饭没 = 2; // 直接编译的话javac会报错，可以用编译工具偷偷改
    if (吃过饭没) System.out.println("吃了");
    if (true == 吃过饭没) System.out.println("真吃了");
  }
}
```
上面这个代码就会只打印第一个，不会打印第二个

**Java的基本类型**

| 类型      | 值域                     | 默认值      | 虚拟机符号 |
| ------- | ---------------------- | -------- | ----- |
| boolean | {falst,true}           | false    | Z     |
| byte    | \[-128, 127\]          | 0        | B     |
| short   | \[-32768, 32767\]      | 0        | S     |
| char    | \[0, 65535\]           | '\u0000' | C     |
| int     | \[-2^31, 2^31 - 1\]    | 0        | I     |
| long    | \[-2^63, 2^63 - 1\]    | 0L       | J     |
| float   | ~\[-3.4E38, 3.4E38\]   | +0.0F    | F     |

其中boolean在Java字节码里可以存储为2，就是读取的时候会有问题。
其他类型也可以存超出值域的数，只不过也会有问题。

**基本类型**
1. 基本类型：boolean、byte、short、char、int、long、float、double
2. 浮点和无穷：
    1. 浮点类型采用IEEE754，有+0.0F即0x0000000和-0.0F即0x8000000，尽管内存数值不同，但+0.0F == -0.0F是true
    2. 任意正浮点（不包括+0.0F）除以+0.0F得到正无穷即0x7F800000，任意正浮点（不包括+0.0F）除以-0.0F得到负无穷即0xFF800000
3. NaN：
    1. [0x7F800001, 0x7FFFFFFF]和[0xFF800001, 0xFFFFFFFF]对应的都是NaN
    2. 一般计算得到的NaN是0x7FC00000，称为标准NaN，其他NaN称为不标准NaN
    3. 对于NaN，除了“!=”始终返回true之外，所有其他比较结果都会返回false 

基本类型的大小
1. 解释器使用的解释栈帧有两个主要部分：局部变量区、字节码的操作数栈
    1. 局部变量区存放：普通意义的局部变量、实例方法的this指针、实例方法的实参
    2. 局部变量区等价于数组，32位jvm的单元长度为4byte，64位jvm的单元长度为8byte，long和double占用两个单元，其他基本类型和引用类型都占一个单元
    3. 以上的小整型占用一个字长的情况仅适用于局部变量，而在堆中的byte/char/short都是1/2/2个byte，即与它们的值域吻合
2. Boolean字段占用1byte，而boolean数组直接用byte数组实现。为保证堆中boolean合法，jvm存储时显式mask掉高位，只取最后一位
3. Jvm中的算数运算几乎都依赖操作数栈，需将堆中的boolean/byte/short/char加载到操作数栈，再在栈上当作int运算
    1. 对于boolean和char，加载时进行零扩展，高位填0
    2. 对于byte和short，加载时进行符号扩展，非负数高位填0，负数高位填1
